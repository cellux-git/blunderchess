# 08 — Pad TT entries to prevent false sharing

**Status**: `wontfix`
**Category**: `enhancement`

> *This was generated by AI during triage.*

TT cache-line padding (ENTRY_SIZE=8) was tested and reverted. With the same 64MB allocation, it cuts entry count 2.67× — TT hit rate drops, nodes increase 2.45×, and total search time increases 65% despite 47% higher NPS. The memory cost to maintain entry count (170MB vs 64MB) doesn't justify the t2 false-sharing benefit. t4 naturally distributes access patterns and performs well without padding.

## Original issue body

## Problem

TT entries are 3 × `AtomicU64` = 24 bytes each. With 64-byte cache lines, ~2.67 entries fit per line. When thread A writes entry N and thread B writes entry N+1 on a different core, they share a cache line — causing false sharing. The line bounces between L1 caches even though the threads access logically independent entries.

## Solution

Pad entries to 64 bytes (one full cache line):

```rust
pub(crate) const ENTRY_SIZE: usize = 8; // 3 used + 5 padding
```

The existing 3 slots stay at indices 0, 1, 2. Slots 3-7 are unused padding. This wastes some memory but the TT is already 64MB — increasing to keep the same entry count would double it. Instead, keep the same total size and halve the entry count (since each entry now takes 2.67× more space).

Alternative: use `#[repr(align(64))]` on buckets:
```rust
#[repr(align(64))]
struct TTBucket([AtomicU64; 3]);
```
Then each bucket occupies its own cache line. The total TT size stays the same, but the number of entries decreases.
